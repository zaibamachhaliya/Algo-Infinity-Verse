🗄️ Why Databases Love B+ Trees

🍃

Data Only In Leaves

Unlike a plain B-Tree, every key's actual data lives in a leaf node. Internal nodes hold only routing keys — no data — so more keys fit per page, keeping the tree shallow.

🔗

Linked Leaves

Every leaf points to the next leaf in sorted order (dashed lines below). This makes range queries like "WHERE id BETWEEN 10 AND 50" a simple linear scan instead of repeated tree descents.

✂️

Splitting On Overflow

When a node exceeds its max key count, it splits in two and pushes a key up to its parent. If the root splits, a brand new root is created — this is the only way a B+ Tree grows taller.

🧩

Borrow Before Merge

When a node shrinks below its minimum, it first tries to borrow a key from a sibling. Only if no sibling can spare one does it merge — keeping the tree balanced with minimal restructuring.
