Vector Clocks Visualizer

No shared clock. No coordinator. Still know exactly what happened before what.

Every node keeps a vector of counters, one per node. A local event increments its own counter. Receiving a message merges clocks by taking the component-wise maximum, then increments. This lets a distributed system correctly detect causality — and just as importantly, correctly detect true concurrency, which no single shared timestamp can do.

Lamport vs Vector Clock — Where Lamport Gets It Wrong

Lamport timestamps compress causality into a single integer — useful for total ordering, but they can't distinguish "A happened before B" from "A and B were concurrent but A's counter happened to end up smaller." Run the scenario below to see this exact failure.

🕸️ The Rules That Make This Work

➕

Local Event Rule

Before any local event, a node increments only its own component in its vector clock. Every other component stays exactly as it was.

🔀

Message Merge Rule

On receiving a message, the receiver's new clock is the component-wise maximum of its own clock and the sender's attached clock, then its own component is incremented once more.

⚖️

The Comparison Rule

Event A happened-before event B if and only if every component of A's clock is ≤ the corresponding component of B's clock, and at least one is strictly less. If neither clock dominates the other, the events are genuinely concurrent.

⚠️

Why This Matters for Conflicts

Riak and the original Dynamo paper use vector clocks precisely to detect when two writes to the same key are truly concurrent (a real conflict needing resolution) versus one simply happening after the other (no conflict, just take the later write).
