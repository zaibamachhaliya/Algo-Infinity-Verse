Mathematical Formula (Rolling Hash)

Rabin-Karp avoids recalculating the hash from scratch. Instead, it subtracts the leaving character's hash contribution, multiplies by the base, and adds the entering character.

$$\text{H}_{i+1} = \Big( d \times \big( \text{H}_i - T[i] \times d^{m-1} \big) + T[i+m] \Big) \pmod q$$

Rabin-Karp Algorithm Core Concepts

Spurious Matches (Collisions)

A **spurious match** (or hash collision) happens when a substring's hash value equals the pattern's hash value, but the actual characters do not match.

Because we use a modulo ($q$) to keep the hash value within integer boundaries, different strings can map to the same hash code ($A \equiv B \pmod q$). If a hash match occurs, we must perform a character-by-character comparison (O(m) verification) to confirm whether it is an actual match or a spurious match.

Rolling Hashing Complexity

Using a rolling hash, we compute the hash of the next window in $O(1)$ constant time. This brings the average search time down to **$O(n + m)$**.

In the worst case (e.g. searching for $A^m$ in $A^n$ with a very poor modulo choice causing collisions at every single window), the algorithm reverts to **$O(n \times m)$** as we verify characters at every step. Choosing a large prime number for $q$ minimizes this risk.
