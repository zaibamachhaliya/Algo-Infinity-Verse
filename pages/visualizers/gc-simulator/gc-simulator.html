Every object tracks how many references point to it. When count hits 0, it's freed immediately. Fatal flaw: cyclic references never reach 0, even when unreachable from root.

📚 Why GC Strategy Matters

⏱️

Mark & Sweep: Stop-the-World

Mark & Sweep must pause your entire program during collection — no allocations or mutations can happen mid-mark, or you'd corrupt the graph. This causes the "GC pause" that makes apps stutter. Used by Java's default collector and Python's cycle detector.

💀

Why Python Added Cycle Detection

CPython uses reference counting as its primary GC — fast, deterministic, immediate. But pure refcounting leaks cycles forever. Python 2.0 added a generational Mark & Sweep specifically to catch cycles that refcounting misses.

📈

The Generational Hypothesis

Empirically, most objects die young — a temporary string, a loop variable, a short-lived closure. Generational GC exploits this: collecting the small young generation frequently is cheap, while the large old generation (likely all alive) is scanned rarely.

🦀

Why Rust Has No GC

Rust's ownership system enforces memory rules at compile time — every value has exactly one owner, freed when the owner goes out of scope. No runtime GC pause, no cycle leaks possible by design (with safe Rust), but the programmer (and compiler) does more work upfront.
