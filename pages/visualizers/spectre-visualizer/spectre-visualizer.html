A performance optimization became the worst security bug in CPU history.

Spectre exploits speculative execution — the CPU's trick of guessing future branch outcomes and executing instructions before it knows if it should. A correct guess means zero wasted cycles. A wrong guess means rollback — but the CPU's cache doesn't roll back. That gap is Spectre.

What is Speculative Execution?

Modern CPUs have a 15–20 stage pipeline. An if branch in the middle means the CPU doesn't know which instructions come next until the condition evaluates — but evaluation takes several cycles. Instead of stalling, the CPU guesses and executes speculatively.

1 CPU fetches the branch instruction and sends it to the branch predictor
2 Predictor says "Taken" — CPU speculatively executes the if-body
3 Branch condition finally resolves: prediction was correct → commit results
4 If wrong: architectural state rolled back, cache NOT rolled back

Training the Branch Predictor

Spectre v1 is a two-phase attack. Phase 1: the attacker trains the branch predictor to always predict "taken" for a specific branch — by calling the victim function repeatedly with in-bounds indices. The predictor builds up state that says "this branch is always taken".

Victim function (C)
if (x < array1_size) {          ← attacker trains this
    y = array2[array1[x] * 512]; ← this is what matters
}

Speculative Out-of-Bounds Read

Phase 2: the attacker calls the victim with a malicious out-of-bounds index that points at secret memory. The bounds check (x < array1_size) will fail — but the predictor, trained to always predict "taken", speculatively executes the body before the bounds check resolves.

🔄 Attacker passes malicious index x = secret_addr - array1_base
⚡ Predictor says "taken" → CPU speculatively reads array1[x] = SECRET_BYTE
🧮 CPU computes array2[SECRET_BYTE × 512] — loads that cache line speculatively
↩️ Bounds check resolves: x is out of bounds → rollback. Registers cleared. Cache line stays.
Mitigation ON: LFENCE inserted after bounds check. CPU stalls until check resolves before executing body speculatively. Cache footprint: none.
Cache State After Speculation
Cache Timing Attack — Recovering the Secret Byte

The attacker now has no architectural evidence of anything. But one cache line from array2 is hot. The attacker times access to array2[i × 512] for every possible byte value i=0..255. The one that hits the cache answers in ~5 cycles instead of ~200 cycles. That i is the secret byte.

(65 = 'A')
Mitigation ON: No speculative read occurred → no cache line was loaded → all 256 accesses measure ~200 cycles (cache miss). Secret byte cannot be determined.

🔬 Understanding Spectre

⚡

Why CPUs Speculate

A modern CPU pipeline takes 15–20 cycles to execute one instruction. A branch stall means 15+ wasted cycles per if. With speculative execution, those cycles are used productively. A correctly-predicted branch has zero stall cost — and most branches are predictable (loop counters, null checks, etc.).

💀

Why Rollback Isn't Enough

The CPU's architectural state (registers, flags, memory writes) is indeed rolled back. But the cache is a physical reality — cache lines that were loaded during speculation remain loaded. This microarchitectural state leaks information that should never have been accessible.

🛡️

Mitigations and Their Cost

LFENCE serializes the pipeline — the CPU must wait for all preceding instructions to complete before speculating past it. This eliminates the attack window but costs 5–30% performance in affected workloads. Retpoline replaced indirect branches with return sequences. These mitigations are why your CPU is slower than it was in 2017.

🌍

Scope of Impact

Spectre v1 affects every CPU with speculative execution — Intel, AMD, ARM, Apple Silicon. Virtually every modern processor. Unlike Meltdown (kernel/user boundary), Spectre attacks can cross process boundaries and is fundamentally harder to fix without hardware changes. Spectre variants are still being discovered in 2024.
