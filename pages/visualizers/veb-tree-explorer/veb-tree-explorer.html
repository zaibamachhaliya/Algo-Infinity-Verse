Van Emde Boas Tree Explorer

Not O(log n). O(log log U). A genuinely different kind of tree.

Every comparison-based tree — BST, AVL, Red-Black, B+ — pays O(log n) per operation. A vEB tree exploits the fact that keys are bounded integers, recursively splitting the universe into √U clusters. For a 16-element universe, that's just 2 levels of recursion instead of 4 comparisons.

📡 How O(log log U) Actually Works

🔀

Splitting the Universe

A vEB tree of universe size u recursively splits into √u clusters, each handling √u values, plus a "summary" tracking which clusters are non-empty. high(x)=⌊x/√u⌋ picks the cluster; low(x)=x mod √u picks the position within it.

⚡

The Lazy Min/Max Trick

Every node caches its own min and max directly — they are NOT stored recursively inside clusters. Insert first checks: "is this the new min?" If a cluster was empty, only its summary needs a single recursive insert — the value becomes that cluster's min/max without recursing further.

🎯

Successor via Summary

To find the next element after x: first check if x's own cluster has anything larger. If not, ask the summary structure for the next non-empty cluster — a single recursive call — then return that cluster's min.

🌐

Real-World Use

IP routing tables use vEB-like structures for longest-prefix matching over 32-bit address spaces. Real-time schedulers use them to find the next deadline in O(log log U) instead of O(log n), critical when U is fixed (like a 32-bit timestamp) regardless of how many events are scheduled.
