Introduction to Fenwick Tree / BIT

A Fenwick Tree (also called Binary Indexed Tree or BIT) is a data structure that efficiently supports two operations on an array: point update and prefix sum query — both in O(log n) time and O(n) space.

Key Insight — The lowbit Trick

  • Each index i in the BIT stores the sum of a range determined by the lowest set bit of i
  • lowbit(i) = i & (-i) — isolates the lowest set bit
  • Update: add lowbit(i) to move to next responsible node
  • Query: subtract lowbit(i) to move to parent prefix
JavaScript 
// The core BIT operation
function lowbit(i) {
  return i & (-i); // isolates lowest set bit
}
// lowbit(6) = 110 & 010 = 2
// lowbit(4) = 100 & 100 = 4
// lowbit(5) = 101 & 011 = 1
BIT Responsibilities (1-indexed, n=8)
BIT[1] = sum of arr[1..1]  (lowbit=1)
BIT[2] = sum of arr[1..2]  (lowbit=2)
BIT[3] = sum of arr[3..3]  (lowbit=1)
BIT[4] = sum of arr[1..4]  (lowbit=4)
BIT[5] = sum of arr[5..5]  (lowbit=1)
BIT[6] = sum of arr[5..6]  (lowbit=2)
BIT[7] = sum of arr[7..7]  (lowbit=1)
BIT[8] = sum of arr[1..8]  (lowbit=8)

Why BIT over Prefix Sum Array?

A static prefix sum array answers range queries in O(1) — but breaks the moment you update an element. BIT handles both queries and updates efficiently.

ApproachBuildPoint UpdatePrefix QuerySpace
Naive ArrayO(n)O(1)O(n)O(n)
Prefix Sum ArrayO(n)O(n) — rebuildO(1)O(n)
Fenwick Tree (BIT)O(n log n)O(log n)O(log n)O(n)
Segment TreeO(n)O(log n)O(log n)O(4n)
💡

When to use BIT: Dynamic prefix sum queries where elements get updated. Simpler to code than a segment tree and uses less memory.

Practice: Choose the Right Tool

You have an array where you need 10,000 point updates and 10,000 range sum queries interleaved. Which structure do you use?

Answer
// BIT: both update and query are O(log n)
// 20,000 ops × O(log n) ≈ 20,000 × 17 = 340,000 ops ✅
// Prefix sum: update = O(n)
// 10,000 updates × O(n) = 10,000 × 10,000 = 100M ops ❌

Building a BIT

The BIT array is 1-indexed (index 0 unused). Build it by calling update for each element, or use the optimized O(n) construction.

JavaScript 
class BIT {
  constructor(n) {
    this.n = n;
    this.tree = new Array(n + 1).fill(0); // 1-indexed
  }

  build(arr) {
    for (let i = 0; i < arr.length; i++) {
      this.update(i + 1, arr[i]);
    }
  }
}

// arr = [3, 2, -1, 6, 5, 4, -3, 3]
// tree= [0, 3, 5, -1, 10, 5, 9, -3, 19]

tree[4] = arr[1...4]   = 10 // lowbit=4
04

Point Update

To update index i by delta, propagate the change upward by repeatedly adding lowbit(i) to i.

JavaScript 
update(i, delta) {
  while (i <= this.n) {
    this.tree[i] += delta;
    i += i & (-i); // next responsible node
  }
}

// Update index 3 by +5 (n=8):
// i=3 → tree[3]+=5, i=3+1=4
// i=4 → tree[4]+=5, i=4+4=8
// i=8 → tree[8]+=5, i=8+8=16 > n, stop
⚡

O(log n) updates: The while loop runs at most log₂(n) times because each step adds at least the lowest set bit, so i always moves to a strictly larger index.

Practice: Trace Update

i=8: tree[8]+=3, lowbit=8, i=16 > 8, stop
Nodes updated: tree[5], tree[6], tree[8] ✅

Prefix Sum Query

To query the prefix sum from index 1 to i, collect partial sums by repeatedly subtracting lowbit(i).

JavaScript 
query(i) {
  let sum = 0;
  while (i > 0) {
    sum += this.tree[i];
    i -= i & (-i); // move to parent prefix
  }
  return sum;
}

// query(7): i=7→6→4→0
// sum = tree[7] + tree[6] + tree[4]
// sum = -3 + 9 + 10 = 16 ✅
Query Path Visualization
query(6): 6 → 4 → 0   (reads tree[6], tree[4])
query(7): 7 → 6 → 4 → 0   (reads tree[7], tree[6], tree[4])
query(8): 8 → 0   (reads tree[8] only — covers all!)
Practice: Trace Query

For arr = [1, 2, 3, 4, 5] with BIT built, trace query(4) step by step.

Trace

query(4):
  i=4: sum += tree[4] = 10
  i = 4 - lowbit(4) = 0, stop
  return 10 ✅ // = 1+2+3+4

Range Sum Query

A range sum from l to r is simply query(r) - query(l - 1). The prefix sum property makes this O(log n).

JavaScript 
class BIT {
  constructor(n) {
    this.n = n;
    this.tree = new Array(n + 1).fill(0);
  }
  update(i, delta) {
    for (; i <= this.n; i += i & -i)
      this.tree[i] += delta;
  }
  query(i) {
    let s = 0;
    for (; i > 0; i -= i & -i)
      s += this.tree[i];
    return s;
  }
  rangeQuery(l, r) {
    return this.query(r) - this.query(l - 1);
  }
}
Console Output
bit.rangeQuery(2, 4) → 9   (before update)
bit.rangeQuery(2, 4) → 19  (after update(3, +10))
Practice: Range + Update

For arr = [5, 3, 7, 2, 8]: (a) compute rangeQuery(2, 4), (b) call update(3, -4), (c) compute rangeQuery(2, 4) again.

Solution
(a) rangeQuery(2,4) = 3+7+2 = 12
(b) update(3, -4) → arr[3] = 3
(c) rangeQuery(2,4) = 3+3+2 = 8 ✅

Inversion Count Problem

Count the number of inversions in an array — pairs (i, j) where i < j but arr[i] > arr[j]. BIT solves this in O(n log n).

Algorithm

  • Process elements from right to left
  • For each element v, query BIT for prefix sum of [1, v-1] — counts smaller elements already processed (to the right)
  • Then update BIT at index v
  • Coordinate compress values first if range is large
JavaScript 
function countInversions(arr) {
  const n = arr.length;
  const sorted = [...new Set(arr)].sort((a, b) => a - b);
  const rank = new Map(sorted.map((v, i) => [v, i + 1]));
  const bit = new BIT(sorted.length);
  let inversions = 0;

  for (let i = n - 1; i >= 0; i--) {
    const r = rank.get(arr[i]);
    inversions += bit.query(r - 1); // smaller to right
    bit.update(r, 1);
  }
  return inversions;
}

console.log(countInversions([3, 1, 2])); // 2
Practice: Count Inversions

Answer
// Fully reverse sorted = maximum inversions
// n*(n-1)/2 = 5*4/2 = 10
(5,4),(5,3),(5,2),(5,1),
(4,3),(4,2),(4,1),
(3,2),(3,1),(2,1) // 10 pairs ✅

BIT vs Segment Tree

Both solve similar problems in O(log n). Knowing when to use each is what competitive programmers and interviewers test.

FeatureFenwick Tree (BIT)Segment Tree
Code complexity~10 lines~40 lines
SpaceO(n)O(4n)
Point updateO(log n)O(log n)
Prefix queryO(log n)O(log n)
Range update + point query✅ With difference trick✅ With lazy propagation
Range update + range query✅ With 2 BITs✅ Easier with lazy prop
Min/Max queries❌ Not directly✅ Natively
Use caseSum queries, inversions, competitiveGeneral range queries, min/max
🎯

Rule of thumb: If you only need sum queries and updates — use BIT. If you need min/max or complex lazy propagation — use Segment Tree. BIT is simpler to implement under contest pressure.

Practice: Choose the Right Tool

You need range minimum queries with point updates. BIT or Segment Tree?

Answer
// Segment Tree — minimum is not invertible
// BIT works for invertible ops: sum, XOR, product
// Min/Max queries need Segment Tree (or Sparse Table for static)
// Segment Tree: O(log n) point update + O(log n) range min ✅
